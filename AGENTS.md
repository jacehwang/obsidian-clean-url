# Clean URL project guide

## Project summary

- This repository is an Obsidian community plugin with plugin ID `obsidian-clean-url`.
- Purpose: clean pasted URLs by removing known tracking parameters while preserving surrounding Markdown text.
- Current release target: `main.js` and `manifest.json` at the plugin root.
- Current license: `MIT`.
- This repository lives directly under a vault plugin directory for local testing:
  - `<Vault>/.obsidian/plugins/obsidian-clean-url/`

## Tooling and environment

- Package manager and script runner: `bun`
- Build pipeline: `tsc --noEmit` + `esbuild`
- Tests: `bun test`
- Lint: `bun run lint`
- Dev watch: `bun run dev`
- Production build: `bun run build`

Use these commands:

```bash
bun install
bun run dev
bun run build
bun test
bun run lint
```

Notes:

- Do not reintroduce `npm` or `package-lock.json` unless explicitly requested.
- `main.js` is generated by esbuild. Do not hand-edit it.
- `bun.lock` is the canonical lockfile.

## Current architecture

Source layout:

```text
src/
  main.ts
  settings.ts
  commands/
    register-commands.ts
  core/
    clean-url.ts
    clean-url.test.ts
    replace-urls-in-text.ts
    replace-urls-in-text.test.ts
    tracking-params.ts
    tracking-params.test.ts
    types.ts
  editor/
    register-paste-handler.ts
```

Responsibilities:

- `src/main.ts`
  - plugin lifecycle only
  - load/save settings
  - register commands
  - register paste handler
- `src/settings.ts`
  - settings type and defaults
  - settings tab UI
  - conversion from plugin settings to core options
- `src/core/clean-url.ts`
  - clean a single URL string
  - remove known tracking parameters from the query string only
  - preserve `#fragment` by default
- `src/core/replace-urls-in-text.ts`
  - scan pasted text and replace every supported URL in place
  - preserve surrounding Markdown and punctuation
- `src/core/tracking-params.ts`
  - shared tracking parameter definitions and normalization helpers
- `src/editor/register-paste-handler.ts`
  - handle Obsidian `editor-paste`
  - replace pasted selection only when transformed text differs
- `src/commands/register-commands.ts`
  - register stable user-facing commands

## Product decisions already made

These are intentional and should not be changed casually:

- v1 scope includes cleaning all supported URLs inside pasted text, not just a single pasted URL.
- Markdown links like `[label](https://example.com?utm_source=x)` must be cleaned.
- Autolinks like `<https://example.com?utm_source=x>` must be cleaned.
- Raw URLs inside prose must be cleaned without altering surrounding text.
- `#fragment` is preserved by default.
- Cleaning is conservative:
  - remove only known tracking parameters
  - do not aggressively strip unknown query parameters
- Cleaning currently applies to query parameters only.
  - Example: `?utm_source=x` is cleaned.
  - Example: `#section&utm_source=x` is not cleaned because that is part of the fragment.
- Parsing failures must leave the original text unchanged.
- Operations should remain idempotent.

## Tracking parameter policy

Current built-in removals:

- `utm_*`
- `fbclid`
- `gclid`
- `dclid`
- `mc_cid`
- `mc_eid`
- `igshid`
- `mkt_tok`

User-configurable extra parameter names are supported via settings.
User-configurable preserved parameter names are also supported via settings.
Preserved parameter rules take precedence over built-in and extra removal rules.

Do not add ambiguous parameters such as `ref`, `source`, or `si` to the built-in list without a deliberate product decision. The plugin currently prefers false negatives over breaking URLs.

## Obsidian integration details

- Paste handling uses `app.workspace.on("editor-paste", ...)`.
- Always check `event.defaultPrevented` before handling the paste.
- Only call `event.preventDefault()` when the pasted text actually changes.
- Use `editor.replaceSelection(cleanedText)` after preventing default.
- Keep `main.ts` minimal. New feature logic belongs in separate modules.
- Command IDs are part of the plugin's stable surface. Avoid renaming existing command IDs once released.

## Settings and UX rules

Current settings:

- `enablePasteCleaning`
- `preserveHash`
- `extraTrackingParams`
- `preservedTrackingParams`

UX constraints:

- Use sentence case for settings, notices, and command names.
- Keep copy short and literal.
- Prefer "link" in UI labels over repeating the plugin name.

## Testing expectations

Before finishing any feature work, run:

```bash
bun test
bun run lint
bun run build
```

Important manual checks in Obsidian:

- paste a block with multiple URLs
- paste Markdown links
- paste autolinks
- paste URLs with trailing punctuation
- verify already clean text remains unchanged
- verify invalid URL-like text remains unchanged
- verify `Clean selected links` command works on selected text
- verify `Preserve fragment` setting changes behavior as expected
- verify `Parameters to keep` preserves selected names even when they match built-in or extra removal rules
- verify disabling automatic paste cleaning stops paste interception

## Local manual testing

Because this repo is already inside the vault plugin directory, local testing is simple:

1. Run `bun run dev`
2. Open the target vault in Obsidian
3. Enable **Clean URL** in **Settings → Community plugins**
4. Reload Obsidian after source changes
5. Paste test content into a Markdown editor

Release artifacts must still be the root-level:

- `main.js`
- `manifest.json`

This project does not use `styles.css`. Do not reintroduce it unless the plugin gains real custom UI styling.

## Lint and test caveats

- Keep tests next to the source files they cover, using `*.test.ts`.
- If you change lint configuration, make sure `bun run lint` remains green without weakening useful project rules.

## Versioning and release notes

- Keep `manifest.json` version and `package.json` version aligned.
- `version-bump.mjs` reads the version from `package.json` and updates:
  - `manifest.json`
  - `versions.json`
- GitHub release tags must match the manifest version exactly, with no leading `v`.

## Things to avoid

- Do not add network requests for URL expansion or canonicalization in v1.
- Do not introduce large dependencies for simple parsing or testing.
- Do not change the plugin ID.
- Do not make the cleaner aggressive by default.
- Do not mutate text outside detected URLs.
- Do not strip fragments unless `preserveHash` is disabled.
